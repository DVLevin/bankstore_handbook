üìù Conventional Commits

Use conventional commit format for clear, parseable history:

feat: New feature
fix: Bug fix
docs: Documentation
style: Formatting
refactor: Code restructure
test: Add tests
chore: Maintenance
perf: Performance

üìã Commit Message Format

type(scope): short description

# Examples:
feat(auth): add JWT refresh token endpoint
fix(payment): handle timeout in Stripe webhook
docs(readme): update installation instructions
refactor(users): extract validation to service class
test(auth): add integration tests for login flow

# With body (for complex changes):
feat(api): implement rate limiting

Add Redis-backed rate limiter with:
- 100 requests per minute per user
- 1000 requests per minute per IP
- Custom limits via X-RateLimit headers

Closes TASK-456

üîÑ Git Workflow

1

Create branch from main

git checkout -b feature/TASK-123-description
2

Make atomic commits

git commit -m "feat(scope): description"
3

Push and create PR

git push -u origin feature/TASK-123-description
4

Address review feedback

git commit -m "fix: address review feedback"
5

Squash merge to main

Merge via GitHub "Squash and merge"

‚úÖ PR Submission Checklist

  • All tests passing (pytest, ruff, mypy)
  • Branch is up to date with main
  • PR title follows conventional commits format
  • Description explains the "why" not just "what"
  • ClickUp task linked in description
  • Self-review completed before requesting review
  • No console.logs, debugger statements, or TODOs
  • Screenshots added for UI changes
