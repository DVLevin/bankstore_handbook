Think & Plan

The "Mini-Spec"
üõë

STOP. Do not code yet. Open a temporary todo.md or use Cursor's Plan Mode.

Prompt 
"I need to build [Feature X]. Create a step-by-step implementation plan
in todo.md. Do NOT write code yet. List:
1. Files we will create or modify
2. Logic changes required
3. Dependencies needed
4. Order of implementation"
2

Load Context

Feed the AI

Tag relevant files so the AI doesn't hallucinate. Context is everything.

Prompt 
"We are implementing step 1 from todo.md.

Context files:
@src/services/user_service.py (for service patterns)
@src/models/user.py (for model structure)
@src/api/routes/users.py (for route patterns)

Explain your approach before writing any code."
3

Write the Contract

BDD / Gherkin
‚≠ê

The most critical step. Write test behavior BEFORE implementation. This is what separates pros from amateurs.

Gherkin 
# tests/features/document_upload.feature

Feature: Document Upload
  Scenario: Valid PDF Upload
    Given the user is authenticated
    And they have a PDF file under 5MB
    When they POST to /api/documents/upload
    Then the system should save the file to S3
    And return a 201 status code
    And create a document record in the database

  Scenario: Invalid File Type
    Given the user is authenticated
    And they have a .exe file
    When they POST to /api/documents/upload
    Then return a 400 status code
    And return error message "Invalid file type"
4

Implement Atomically

One Feature. One Task. One File.
Prompt 
"Implement the upload_service.py logic to satisfy the Gherkin scenarios.

Rules:
- Do NOT touch the frontend yet
- Follow the service pattern in @user_service.py
- Include proper error handling
- Add type hints and docstrings"
üí°

Golden Rule: Do not let the AI refactor the entire codebase. Focus on the single atomic task.

Manual "Vibe Check"

AI tests can pass even if the UX is broken
  • Spin up the app locally
  • Click the actual buttons
  • Does it feel right?
  • Does the loading state work?
  • Are error messages helpful?
